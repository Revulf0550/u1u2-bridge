## 1. Финальная схема инвертора — ОБЯЗАТЕЛЬНО ЗАПОМНИТЬ

### Перфоплата — раскладка

Все 3 компонента на одной плате:

- **R2 = 4.7 кΩ** (pull-up) — вертикально
- **R1 = 2.2 кΩ** (base resistor) — вертикально, рядом с R2
- **BC547** NPN транзистор — горизонтально снизу, плоской стороной к верху перфоплаты

5 пайных точек на плате:

| Точка | Что в ней | Внешний провод |
|-------|-----------|----------------|
| **[A]** | верх R2 | синий → OPi pin 1 (3.3V) |
| **[B]** | верх R1 | зелёный → OPi pin 29 (UART TX) |
| **[C]** | низ R2 + коллектор BC547 (левая ножка) | белый → модуль JR socket pin 5 (CRSF) |
| **(·)** | низ R1 + база BC547 (средняя ножка) | — внутренняя, без провода |
| **[D]** | эмиттер BC547 (правая ножка) | чёрный короткий → OPi GND плата |

**Ориентация BC547:** плоской стороной с надписью к верху перфоплаты. Тогда слева направо: **C — B — E**.

### Полная схема — 6 проводов в системе

| # | Провод | Состояние | От | До | Назначение |
|---|--------|-----------|----|----|------------|
| 1 | 🔵 синий | НОВЫЙ | OPi pin 1 (3.3V) | плата [A] | питание pull-up R2 |
| 2 | 🟢 зелёный | МОДИФИЦИРОВАН | OPi pin 29 (UART TX) | плата [B] | сигнал; **R1 в проводе УБРАН** (теперь на плате) |
| 3 | ⚫ чёрный короткий | НОВЫЙ | OPi GND плата | плата [D] | GND эмиттера |
| 4 | ⚪ белый | НОВЫЙ | плата [C] | модуль pin 5 | инвертированный CRSF — ЕДИНСТВЕННЫЙ сигнальный в модуль |
| 5 | 🔴 красный | СТАРЫЙ, не трогать | OPi pin 2 (5V) | модуль pin 3 | питание модуля 5V — мимо платы |
| 6 | ⚫ чёрный длинный | СТАРЫЙ, не трогать | OPi GND плата | модуль pin 4 | общий GND модуля — мимо платы |

**В модуль (TX) идёт ВСЕГО 3 провода:**
- pin 3 ← красный (5V, старый)
- pin 4 ← чёрный длинный (GND, старый)
- pin 5 ← белый (CRSF, новый, через инвертор)

### Логика инвертора

- **UART HIGH (3.3V)** → ток через R1 → база открыта → BC547 проводит → выход [C] прижат к GND через эмиттер → **выход LOW** (0V)
- **UART LOW (0V)** → база закрыта → коллектор подтянут через R2 к 3.3V → **выход HIGH** (3.3V)

Инверсия в обе стороны — как нужно для ESP32-based ELRS Ranger Micro.

### Что изменилось от первоначальной идеи

В прошлой сессии R1 был припаян inline в зелёный провод (с термоусадкой). В этой сессии решили **переместить R1 на плату** — чище и надёжнее, никаких компонентов в проводах. Пользователь должен был:
1. Отрезать зелёный провод с обеих сторон от старого R1, выкинуть кусок с R1.
2. На плату поставить новый 2.2 кΩ.
3. Зелёный провод теперь идёт от OPi pin 29 прямо в [B].

Пользователь подтвердил пайку, но **визуально/мультиметром детали не проверял на этом этапе**. Возможный риск — если что-то не сработает на smoke-тесте, в первую очередь проверять что старый R1 действительно убран.

## 2. UFW Lesson — КРИТИЧЕСКИ ВАЖНО

### Что случилось

После пайки пользователь попытался подключиться к u2 для smoke-теста:
- `ssh ubuntu@10.8.0.7` — не работал (WireGuard упал, "временный сбой")
- `ssh ubuntu@192.168.31.100` — `Connection timed out`

### Диагностика

Через физический доступ (HDMI + USB клавиатура к u2) запустили `sudo ufw status verbose`:

```
Status: active
Default: deny (incoming), allow (outgoing), disabled (routed)

To           Action      From
--           ------      ----
22/tcp       ALLOW IN    10.8.0.5
14550/udp    ALLOW IN    10.8.0.5
14551/udp    ALLOW IN    10.8.0.5
14550/udp    ALLOW IN    10.8.0.6      # u1-pi CRSF1
14551/udp    ALLOW IN    10.8.0.6      # u1-pi CRSF2
```

Пакеты с 192.168.31.150 (ноут) → не подпадают ни под одно правило → default `deny` → **silent drop** → `Connection timed out` (НЕ `refused` — это диагностически важно).

### Решение применённое

```bash
sudo ufw allow from 192.168.31.0/24 to any port 22
```

После этого `ssh ubuntu@192.168.31.100` с Windows заработал.

### Правило к запоминанию навсегда

**На u2-pi UFW настроен пропускать SSH (порт 22) только из:**
- `10.8.0.5/32` — ноут пользователя через WireGuard (основной путь)
- `192.168.31.0/24` — домашняя LAN (fallback, добавлено 2026-05-23)

**Lesson:** на production-устройстве с UFW и VPN — всегда оставлять fallback-доступ из локальной сети. Иначе при отказе VPN устройство становится unreachable удалённо, и нужен физический доступ для починки.

```bash
sudo python3 << 'PYEOF'
import serial, time
def crc8(data, poly=0xD5):
    crc = 0
    for byte in data:
        crc ^= byte
        for _ in range(8):
            crc = ((crc << 1) ^ poly) & 0xFF if crc & 0x80 else (crc << 1) & 0xFF
    return crc
def crsf_rc_packet():
    channels = [992] * 16
    payload = bytearray(22)
    bit_idx = 0
    for ch in channels:
        for b in range(11):
            if ch & (1 << b):
                payload[bit_idx // 8] |= (1 << (bit_idx % 8))
            bit_idx += 1
    body = bytes([0x16]) + bytes(payload)
    return bytes([0xC8, 24]) + body + bytes([crc8(body)])
ser = serial.Serial('/dev/ttyS7', 420000)
pkt = crsf_rc_packet()
print(f'>>> Sending CRSF, hex={pkt.hex()}')
n = 0
start = time.time()
try:
    while True:
        ser.write(pkt)
        ser.flush()
        n += 1
        time.sleep(0.004)
        if n % 500 == 0:
            print(f'>>> {n} packets, {n/(time.time()-start):.0f} Hz', flush=True)
except KeyboardInterrupt:
    print(f'\n>>> Stopped after {n} packets')
finally:
    ser.close()
PYEOF
```

Параллельно — телефон/ноут сканирует WiFi-сети. Ждать 60+ секунд:

- **WiFi `ExpressLRS TX` НЕ появляется** → 🎉 инверсия решила проблему. Идти дальше к telemetry/back-channel и интеграции.
- **WiFi `ExpressLRS TX` появляется** → инверсия не помогла. Глубокая диагностика: либо мисмонтаж (мультиметром прозвонить плату), либо проблема не в инверсии (возможно signal integrity, нужен осциллограф).
