*(истина — репо/скрипты; ниже — сжато для контекста)*

**CRSF:** Boxer → u1 `/dev/ttyUSB0` (CH340 `1a86:7523`, прямое kernel-имя — для одного адаптера ОК, см. урок 2026-05-27) → `crsf-bridge@p1` → UDP `:14552` → wg → `crsf-bridge@elrs` → u2 `/dev/ttyS7` (аппаратный UART7) → ELRS-TX → RF → дрон.

**Видео TX (u2)** — `video_tx.sh`: `v4l2src /dev/video0 ! image/jpeg,640x480,30/1 ! jpegdec ! videoconvert ! NV12 ! mpph264enc rc-mode=cbr bps=2500000 gop=15 profile=66 ! h264parse ! rtph264pay pt=96 mtu=1200 ! udpsink`.
Грабер: **Arkmicro `18ec:5555`, ТОЛЬКО 640×480 MJPG@30, БЕЗ `io-mode=4`**.

**Видео RX (u1)** — `video_rx.sh` под `cage` от root: `udpsrc :5600 ! rtpjitterbuffer latency=50 ! rtph264depay ! h264parse ! mppvideodec ! videoconvert ! waylandsink sync=false`.
`JITTER_LATENCY=50`, `mtu=1200` — тюнились под интернет-WG 180 мс. **Под радио (3–7 мс) можно опустить latency** (порог дропов ранее находили ~10 мс). `/run/user/0` (0700 root) пересоздаётся tmpfiles ДО старта `video-rx`.

## ПЛАН ДНЯ

**2b — физический cutover (провода):**
- Pi-u1 → порт `LAN` PoE-адаптера **CPE-AP**; Pi-u2 → порт `LAN` PoE-адаптера **CPE-Client**; CPE-коробки — только в порт `POE` своих адаптеров.
- **Pi должны быть на ПРОТИВОПОЛОЖНЫХ концах радио** — иначе трафик уйдёт по проводу и радио не проверишь.
- **ARDOR остаётся в домашнем свитче** (сохраняет интернет/ассистента). **После cutover Pi теряют WG → SSH к Pi с ARDOR пропадёт.** Доступ к Pi: монитор u1, либо ARDOR dual-home (Wi-Fi=интернет + Ethernet=мост для SSH).

**2c — проверка по радио:**
- Видео: живая картинка на мониторе u1 (⚠️ при выкл. дроне синий no-signal и фриз НЕОТЛИЧИМЫ → проверять с дроном/движением, либо считать RTP / `ping .20↔.10` по радио с бенча).
- CRSF: реакция дрона на стики. **Дрон включать — ТОЛЬКО спросив пользователя ЯВНО.**
- Перетюнить `JITTER_LATENCY` под радио.

**2d — финализация в репо:**
- CRSF: `sudo TRANSPORT=direct SKIP_NETPLAN=1 ./install.sh u1|u2`.
- **Этап D** — видео-peer в Режим №2 (сейчас peer видео живёт в `video_tx.sh` через systemd-override, install.sh его НЕ покрывает — это пробел).

## УРОКИ ЭТОЙ СЕССИИ (зафиксировать на 2d)
1. **UFW + смена источника = тихий дроп.** При репойнте любого CRSF/видео-peer на новый источник-IP надо открыть UFW на ПРИЁМНИКЕ под этот источник, иначе пакеты молча дропаются. Проверять ФАКТ потока (`udp->uart` / счёт RTP), а НЕ `is-active`. (Совпадает с уроком 2026-05-24.)
2. **Не верить маленькому tcpdump-сэмплу.** `-c 10` за ~100 мс случайно попал в межкадровый промежуток (23-байтные пакеты) → ложная тревога «видео мёртвое». 5-сек гистограмма размеров опровергла.
3. **CBR маскирует no-signal.** ~2.5 Mbps идут и на статичной картинке — здоровый поток ≠ живой контент.

## ❗ НЕ ДЕЛАЙ (carry-over + новое)
- **НЕ редеплоить `install.sh` без `TRANSPORT=… SKIP_NETPLAN=1`** — иначе CRSF env вернётся к `10.8.0.x` И install.sh перезапишет netplan → конфликт.
- **НЕ добавлять secondary-IP на Ethernet ARDOR при активном на нём интернете** — рвёт интернет (проверено). На cutover: интернет ARDOR по Wi-Fi, Ethernet — отдельно на мост (без gateway).
- **Pi на cutover — на разные концы радио.**
- НЕ `kmssink` (VOP2 timeout) — только `cage`+`waylandsink`. НЕ убирать `videoconvert` перед `waylandsink` (чёрный на NV12). Грабер — только 640×480 MJPG, без `io-mode=4`.
- НЕ доверять старому `/opt` без редеплоя. `pyproject.toml` — чужое. `docs/baseline/` — не трогать.
- **ДРОН включать — ТОЛЬКО спросив пользователя ЯВНО.**
- git `-m`: трейлеры вторым `-m`. Длинные файлы — через `present_files`, не вставкой в терминал.
- В Claude Code Bash-тулзе НЕ использовать `cd ~/...` (резолвится в Windows-домашку; cwd и так корень репо).
- Запись в CLAUDE.md/память про `192.168.1.x`: формулировать ТОЧНО — это `.x` вторым IP на домашнем свитче (L2), **CPE710 в путь Pi ещё НЕ вставлен**. Не записывать «CPE развёрнут».
