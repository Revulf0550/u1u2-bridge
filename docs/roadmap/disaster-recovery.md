# Disaster recovery — быстрое восстановление Pi после отказа

> Трек: восстановление узла после отказа (умер накопитель / система не грузится).
> Смежно, но **иной класс**, чем `production-hardening.md`: там — устойчивость
> работающего устройства (RO-rootfs, watchdog, OSD); здесь — что делать, когда
> устройство уже отказало.

## Статус

Стратегия выбрана (2026-06-25).
- **Уровень 1** (воспроизводимость) — можно делать сейчас.
- **Уровень 2** (полный клон) — завершающий этап проекта.

---

## Принцип

Бо́льшая часть системы уже воспроизводима **без образа**:
- код проекта — в git (origin), `git clone`;
- деплой детерминирован — `install.sh u1|u2` ставит сервисы, env, udev, systemd, sysctl;
- ОС — Ubuntu (Joshua Riek), прошивается на чистый носитель за минуты.

Бэкапить надо **только то, что `install.sh` НЕ покрывает** или генерирует с плейсхолдерами.

---

## Уровень 1 — бэкап уникального состояния (воспроизводимость, СЕЙЧАС)

**Сценарий восстановления:** чистая ОС → `git clone` → `install.sh u1|u2` → накат бэкапа поверх.

**Что бэкапить с каждой Pi** (то, чего нет в репо или сгенерено с плейсхолдерами):
- `/etc/wireguard/wg0.conf` + ключи. Приватный ключ узла **невосстановим** — только
  перегенерация + повторная передача публичного ключа пиру.
- живые `/etc/u1u2-bridge/crsf-*.env` — особенно если правлены руками после `install.sh`
  (фактический `PEER=` / режим транспорта на узле может отличаться от того, что
  сгенерил `install.sh` при деплое; живое значение есть только на узле).
- `/etc/udev/rules.d/90-u1u2-uart.rules` с реальными серийниками адаптеров.
- netplan-статика (`/etc/netplan/99-*.yaml`).
- любые ручные правки вне репо.

**Объём** — мегабайты текста. Не требует WG off (проходит даже через туннель).

## Уровень 2 — полный посекторный клон («воткнул и летит», ЗАВЕРШАЮЩИЙ ЭТАП)

Делать, **когда проект закрыт по всем вопросам** (грабер cx231xx устаканен, трек D
определён). Раньше нет смысла — клонировать то, что ещё поменяется.

**Метод:** `dd` по **прямому пути** (WG off → гигабит по свитчу End-A) либо локально на
внешний носитель, подключённый к Pi.

**Не через туннель:** RTT туннеля ~210 мс (замер 2026-06-25) для полного образа
непригоден — десятки/сотни ГБ через публичный endpoint, обрыв handshake обнуляет `dd`.

---

## Бэклог (QoL-доступ, НЕ hardening)

Рассмотреть сужение `AllowedIPs` (исключить `192.168.1.0/24` из туннеля) для
одновременного прямого + туннельного доступа с ARDOR.
- Предусловие выполнено: ARDOR на LAN моста (подтв. 2026-06-25, общий свитч с u1/u2).
- **Ослабляет kill-switch** (сейчас `0.0.0.0/0` — подтв. живой `wg show` 2026-06-25) —
  решать по модели угроз.
- Требует: живое чтение `.conf`, бэкап профиля, проверку маршрутов (радиолинк vs туннель).
