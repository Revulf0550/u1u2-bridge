# Пульт №2 — USB-HID

## Статус
**БУДЕМ ДЕЛАТЬ.** Зафиксированный план B (не отброшен). Инфраструктура Пульта №1
(CRSF через JR-bay) подтверждена LIVE end-to-end (CRSF до дрона, ARM работает),
поэтому к варианту №2 возвращаемся как к следующему запланированному шагу.
Решение пользователя: «Готов на +10 мс, Lua не использую, применим обязательно, но позже».

## Суть
Boxer (или любой USB-пульт/джойстик/gamepad) подключается напрямую USB-кабелем к
u1-pi как HID-устройство — нулевая пайка, нет риска спалить UART.

```
Boxer/любой gamepad USB-C → u1-pi (HID joystick mode)
  → joystick_to_udp.py: читаем evdev /dev/input/jsX, пакуем 16 каналов в UDP
  → u2-pi: joystick_to_crsf.py распаковывает UDP
  → генерирует CRSF frame (sync 0xC8, type 0x16, channel data 22 байта, CRC8)
  → UART7 → SN74HC14N → ELRS Ranger Micro → дрон
```

**Плюсы:** нулевая пайка; универсальность — любой USB-gamepad (Boxer, Xbox-контроллер
и т.п.) заменяем в поле; нет риска спалить UART; пульт тестируется как обычный gamepad
на любом ПК.

**Минусы:** +10 мс латенции (USB-HID poll 125–1000 Hz + кодирование CRSF на u2);
не работают Lua-скрипты и TX menu; telemetry обратно на пульт невозможна простыми
средствами.

**Объём работы:** дописать `joystick_to_udp.py` (u1) + `joystick_to_crsf.py` (u2),
~4–6 ч.

## Когда
После того как Пульт №1 доказал работоспособность инфраструктуры (уже доказал —
LIVE). Можно браться следующим, либо раньше, если +10 мс окажутся критичны/некритичны
по полевым результатам.
